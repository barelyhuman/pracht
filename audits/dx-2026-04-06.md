# DX Audit

Date: 2026-04-06

## Verdict

Promising framework design, weak current execution DX.

Roughly:

1. Authoring DX: `7/10`
2. Setup/build DX: `3/10`
3. Production confidence DX: `2/10`
4. Overall today: `4/10`

This reads more like a strong internal scaffold than a framework that feels ready for outside users.

## What Works Well

1. The core authoring model is clean and readable. The manifest API in `examples/basic/src/routes.ts` is explicit without being too noisy.
2. Route modules are easy to understand. `loader` + `Component` + shell composition in the example app is straightforward and nicely typed.
3. Package boundaries are sensible: framework, Vite plugin, adapter, CLI.
4. The basic dev SSR loop is real enough to exercise. `pnpm test`, `pnpm e2e`, and `pnpm typecheck` all passed.

## Main DX Problems

1. The documented production path is broken.
   - `pnpm build:example` fails.
   - Root cause: packages export raw TypeScript source, for example `packages/vite-plugin/package.json` and `packages/framework/package.json`.
   - That makes consumption fragile and currently breaks Vite loading the plugin from the example.

2. `viact dev` is not actually "one command" in practice.
   - Running the example dev server directly fails unless Node is started with `--experimental-strip-types`.
   - Playwright only works because `playwright.config.ts` injects that workaround.
   - That directly contradicts the "instant local DX / zero config" goal in `VISION_MVP.md`.

3. Client-side navigation DX is mostly not there yet.
   - The generated client module in `packages/vite-plugin/src/index.ts` only hydrates the current route.
   - There is no link interception, history handling, or client router state update.
   - `docs/WORKSPACE.md` already calls this out as the biggest user-facing gap.

4. Forms and revalidation are currently underspecified in behavior.
   - `Form` is just a plain form element wrapper in `packages/framework/src/runtime.ts`.
   - `useRevalidateRoute()` and `useSubmitAction()` only `fetch()` and return data; they do not update route state or re-render the app.
   - The docs in `docs/DATA_LOADING.md` promise more than the runtime currently delivers.

5. SSG/ISG are positioned as framework capabilities, but the build pipeline does not implement them yet.
   - `packages/cli/bin/viact.js` only runs client and server Vite builds.
   - There is no prerender step, no static HTML emission, no ISG regeneration loop.
   - `docs/WORKSPACE.md` confirms these are still next steps.

6. Error-handling DX is incomplete.
   - `ViactHttpError` and `ErrorBoundary` exist in `packages/framework/src/types.ts`.
   - `handleViactRequest()` in `packages/framework/src/runtime.ts` never catches those errors and never renders an error boundary.
   - Action envelopes with `redirect` or `headers` are also just JSON-serialized, not translated into actual HTTP behavior.

7. Docs trust is currently a DX problem.
   - `VISION_MVP.md` claims Phase 1 already proves SSG and client-side navigation.
   - `docs/WORKSPACE.md` says those are still upcoming.
   - Clear docs are good; inaccurate docs are worse than sparse docs for framework DX.

## Smaller Papercuts

1. There is no root `README` or quickstart. Discovery is contributor-oriented rather than user-oriented.
2. `pnpm test` ran the same router tests multiple times through workspace-linked `node_modules`, which is noisy contributor DX.
3. The Node experimental warning shows up in normal flows, which makes the toolchain feel provisional.

## Best Parts Of The Design

1. The explicit manifest is a good tradeoff if the target audience values auditability over magic.
2. Preact-first + Vite-native is a strong positioning choice.
3. The example route API is small enough to learn quickly.

## Priority Fix Order

1. Stop exporting raw `src/*.ts` from packages; build packages and export `dist`.
2. Make `viact dev`, `build`, and `preview` work without special Node flags.
3. Either implement or clearly mark missing features: client router, `<Form>` interception, revalidation, SSG, ISG.
4. Add a real root quickstart.
5. Tighten docs so shipped behavior and aspirational behavior are clearly separated.

## Bottom Line

The conceptual DX is good. The actual DX breaks at the exact places framework users care most about: install, dev startup, production build, and trust in the docs.
