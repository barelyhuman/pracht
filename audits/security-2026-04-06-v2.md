## Resolved Since v1

### 1. RESOLVED (was High): XSS via unescaped hydration state

- The hydration state is now serialized into a `<script id="viact-state" type="application/json">` tag using `serializeJsonForHtml()` (`runtime.ts:737,807-814`).
- `type="application/json"` prevents browser execution.
- `serializeJsonForHtml` escapes `<`, `>`, `&`, U+2028, and U+2029, preventing script-block breakout.
- The client reads via `element.textContent` + `JSON.parse` (`runtime.ts:124-140`), which is the safe consumption pattern.
- **Status: Fixed correctly.**

### 2. RESOLVED (was High): No CSRF enforcement for actions

- `validateActionCsrfRequest()` (`runtime.ts:816-828`) now runs before any action handler.
- It checks `Origin` first, then `Referer`, and rejects requests where neither header is present or matches the request origin.
- The `isSameOrigin` helper (`runtime.ts:830-835`) correctly parses both values as URLs and compares `.origin`.
- **Status: Fixed correctly.** Standard Origin/Referer CSRF protection as recommended by OWASP.

### 3. RESOLVED (was High): API routes bypass middleware

- API route dispatch (`runtime.ts:289-304`) now resolves middleware files from `app.api.middleware` and calls `runMiddlewareChain()` before invoking the API handler.
- Middleware can short-circuit with a `Response` or augment context, same as page routes.
- **Status: Fixed correctly.**

### 4. RESOLVED (was Medium): No default browser security headers

- `applyDefaultSecurityHeaders()` (`runtime.ts:767-788`) sets `Permissions-Policy`, `Referrer-Policy`, `X-Content-Type-Options: nosniff`, and `X-Frame-Options: SAMEORIGIN` on all framework-generated responses.
- `withDefaultSecurityHeaders()` wraps every response path in `handleViactRequest`.
- The CLI preview server also applies these via `setDefaultSecurityHeaders()` (`viact.js:362-369`).
- E2E test at `basic.test.ts:31-37` asserts three of these headers.
- **Status: Fixed correctly.** Note: no default `Content-Security-Policy` is set (see new finding below).

## Open Findings

### 5. Medium (unchanged): The example app accepts any `session=` cookie as authenticated.

- References: `examples/basic/src/middleware/auth.ts:3-8`
- Still `request.headers.get("cookie")?.includes("session=")`.
- No signature check, no expiry, no secure cookie attributes guidance.
- This is the pattern users will copy from the starter app.
- **Recommendation:** Replace with a signed-session example using HMAC or clearly label with a `⚠️ NOT FOR PRODUCTION` comment that explains what's missing (signature, HttpOnly, Secure, SameSite, expiry).

### 6. Medium (unchanged): Host and protocol derived from untrusted request headers.

- References: `adapter-node/src/index.ts:161-163`, `vite-plugin/src/index.ts:299-304`, `cli/bin/viact.js:312-314`
- `x-forwarded-proto` and `host` headers are used directly to construct `Request` URLs.
- In a misconfigured proxy chain, this enables host-header poisoning of `url.origin` in route args, which flows into CSRF origin comparison, redirects, and any user code that reads `args.url`.
- **Recommendation:** Document a trust-proxy model. In production adapters, either require explicit configuration of trusted origins or validate `X-Forwarded-*` against a known proxy list.

### 7. Medium (unchanged): Request bodies fully buffered with no size limit.

- References: `adapter-node/src/index.ts:205-218`, `vite-plugin/src/index.ts:318-325`
- Both `readRequestBody` implementations read the full body into memory with no cap.
- A single large POST can consume unbounded memory.
- **Recommendation:** Add a configurable `maxBodySize` (default 1 MB or similar) and reject requests exceeding it with `413 Payload Too Large`.

### 8. Medium (unchanged): `viact preview` has a path traversal risk in static file serving.

- References: `cli/bin/viact.js:283-284`
- `const filePath = join(clientDir, url)` uses the raw request URL. `join()` resolves `..` segments, so a request like `GET /../../package.json` produces a path outside `clientDir`.
- The `existsSync(filePath) && statSync(filePath).isFile()` check then serves it.
- **Recommendation:** After joining, verify that the resolved path starts with `clientDir + "/"` before serving. For example:
  ```js
  const resolved = resolve(clientDir, '.' + parsedUrl.pathname);
  if (!resolved.startsWith(clientDir + '/') && resolved !== clientDir) return next();
  ```

### 9. Low (unchanged): GitHub Actions use floating third-party action tags.

- References: `.github/workflows/release.yml:19-38`, `.github/workflows/ci.yml:16-23`
- Actions pinned to `@v4`, `@v1`, `@v1.5.3` — not commit SHAs.
- The release workflow has `contents: write`, `pull-requests: write`, and `id-token: write`.
- **Recommendation:** Pin to full SHAs. Use `dependabot` or `renovate` to keep them updated.

## New Finding

### 10. Low (new): No default Content-Security-Policy header.

- References: `runtime.ts:767-788`
- The new security headers are a strong baseline, but no `Content-Security-Policy` is set.
- Without CSP, if a future XSS vector is introduced (e.g. user-controlled HTML in a component), there is no defense-in-depth layer to block inline script execution.
- The framework already uses `type="application/json"` for hydration state (good), but the client entry is loaded via `<script type="module" src="...">`, which would be compatible with a strict CSP.
- **Recommendation:** Consider adding a default restrictive CSP or providing a first-class `csp` option in the head metadata API. At minimum, document how users should configure CSP for their Viact apps.

## Coverage Gaps (updated)

| Area | v1 status | Current status |
|------|-----------|----------------|
| XSS-safe serialization of hydration state | No tests | Partially covered — `basic.test.ts:22-29` asserts the `application/json` tag exists but does not test hostile payloads |
| CSRF rejection for action requests | No tests | No tests |
| API routes inherit middleware | No tests | No tests |
| Static-file path traversal in preview | No tests | No tests |
| Default security headers | No tests | Covered — `basic.test.ts:31-37` |
| Request body size limits | No tests | No tests (feature not implemented) |

**Recommendation:** Add focused unit/integration tests for:
1. CSRF: a cross-origin POST to an action route returns 403.
2. CSRF: a same-origin POST to an action route succeeds.
3. Hydration state: a loader returning `</script><script>alert(1)</script>` is safely escaped in the HTML.
4. Preview traversal: a request with `../` segments does not serve files outside `dist/client`.
5. API middleware: an API route behind middleware returns the middleware's response when unauthenticated.

## Priority Fix Order

1. **Fix preview static-file path normalization** (finding 8) — concrete file disclosure bug, simple fix.
2. **Add request-body limits** (finding 7) — unbounded memory consumption on POST.
3. **Improve example auth** (finding 5) — users will copy this; at minimum add a warning comment.
4. **Document or enforce host trust model** (finding 6) — interacts with CSRF validation correctness.
5. **Add CSRF and XSS regression tests** — protect the fixes that were already made.
6. **Pin GitHub Actions to SHAs** (finding 9) — low urgency but easy to do.
7. **Consider default CSP** (finding 10) — defense in depth.

## Bottom Line

The three most critical issues from v1 (XSS, CSRF, middleware bypass) are all properly fixed. The framework now has a real security header baseline and a working CSRF defense. The remaining issues are real but lower severity — the path traversal in preview and unbounded body reads are the most actionable. With those fixed and some test coverage added for the security-critical paths, Viact would be in reasonable shape for production use behind a properly configured reverse proxy.
